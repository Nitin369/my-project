\doxysubsection{Detailed Description}
The Object Request Broker, or \mbox{\hyperlink{classPoco_1_1RemotingNG_1_1ORB}{O\+RB}}, is the central object in the Remoting machinery. The \mbox{\hyperlink{classPoco_1_1RemotingNG_1_1ORB}{O\+RB}} is a singleton which manages \mbox{\hyperlink{classPoco_1_1RemotingNG_1_1Listener}{Listener}}, \mbox{\hyperlink{classPoco_1_1RemotingNG_1_1RemoteObject}{Remote\+Object}} and \mbox{\hyperlink{classPoco_1_1RemotingNG_1_1Skeleton}{Skeleton}} instances on the server side, as well as \mbox{\hyperlink{classPoco_1_1RemotingNG_1_1Proxy}{Proxy}} creation on the client side.

A client can use \mbox{\hyperlink{classPoco_1_1RemotingNG_1_1ORB_afe4cdad854d6c4ebd936b7017d0455fb}{find\+Object()}} to obtain an interface for accessing a service object. Depending on whether the service object has been registered on the same \mbox{\hyperlink{classPoco_1_1RemotingNG_1_1ORB}{O\+RB}}, or not, the \mbox{\hyperlink{classPoco_1_1RemotingNG_1_1ORB}{O\+RB}} will either return a \mbox{\hyperlink{classPoco_1_1RemotingNG_1_1RemoteObject}{Remote\+Object}} (with forwards calls locally, without the need for serialization/deserialization), or a \mbox{\hyperlink{classPoco_1_1RemotingNG_1_1Proxy}{Proxy}}.

It is recommended to use the various helper classes (client helper, server helper) generated by the Remoting code generator to obtain an instance of an object instead of using \mbox{\hyperlink{classPoco_1_1RemotingNG_1_1ORB_afe4cdad854d6c4ebd936b7017d0455fb}{find\+Object()}} directly.

Depending on whether the service object has been registered on the same \mbox{\hyperlink{classPoco_1_1RemotingNG_1_1ORB}{O\+RB}}, or not, the \mbox{\hyperlink{classPoco_1_1RemotingNG_1_1ORB}{O\+RB}} will either return a \mbox{\hyperlink{classPoco_1_1RemotingNG_1_1RemoteObject}{Remote\+Object}} (with forwards calls locally, without the need for serialization/deserialization), or a \mbox{\hyperlink{classPoco_1_1RemotingNG_1_1Proxy}{Proxy}}.

The \mbox{\hyperlink{classPoco_1_1URI}{U\+RI}} must have the following format\+: $<$scheme$>$\+://$<$authority$>$/$<$protocol$>$/$<$type\+Id$>$/$<$object\+Id$>$

This method is usually not called directly, but indirectly through the generated client helper class instead. The client helper class performs an appropriate cast to the actual interface class.

The strategy for determining whether the \mbox{\hyperlink{classPoco_1_1URI}{U\+RI}} references a local object is as follows\+: First, check whether the \mbox{\hyperlink{classPoco_1_1URI}{U\+RI}} matches one of the U\+R\+Is registered with the local \mbox{\hyperlink{classPoco_1_1RemotingNG_1_1ORB}{O\+RB}}. If so, return the matching \mbox{\hyperlink{classPoco_1_1RemotingNG_1_1RemoteObject}{Remote\+Object}} instance. If no matching \mbox{\hyperlink{classPoco_1_1URI}{U\+RI}} is found, ask all registered \mbox{\hyperlink{classPoco_1_1RemotingNG_1_1Listener}{Listener}} instances whether they \char`\"{}know\char`\"{} the given \mbox{\hyperlink{classPoco_1_1URI}{U\+RI}}. To do this, first the protocol, Type\+Id and Object\+Id are extracted from the \mbox{\hyperlink{classPoco_1_1URI}{U\+RI}}. Then, each \mbox{\hyperlink{classPoco_1_1RemotingNG_1_1Listener}{Listener}} handling the protocol is asked to create an \mbox{\hyperlink{classPoco_1_1URI}{U\+RI}} for Type\+Id and Object\+Id. If this \mbox{\hyperlink{classPoco_1_1URI}{U\+RI}} matches one of the registered U\+R\+Is, check whether the respective \mbox{\hyperlink{classPoco_1_1RemotingNG_1_1Listener}{Listener}} actually handles the endpoint contained in the \mbox{\hyperlink{classPoco_1_1URI}{U\+RI}} (by calling \mbox{\hyperlink{classPoco_1_1RemotingNG_1_1Listener_a6626c744e63983b38069a8d7f5f05658}{Listener\+::handles\+U\+R\+I()}}).
